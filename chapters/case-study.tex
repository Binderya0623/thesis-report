\subsection{JavaServer Faces (JSF) технологийн ажиллах зарчим}

JavaServer Faces (JSF) нь Java EE стандартын нэг хэсэг бөгөөд вэб аппликэйшний хэрэглэгчийн интерфэйсийг бүрэлдэхүүн хэсэгт суурилан хөгжүүлэх боломжийг олгодог серверийн талын фрэймворк юм. Хөтөч дээр ажилладаг орчин үеийн SPA архитектураас ялгаатай нь JSF нь серверийн талд хөрвүүлэлт хийх үндсэн зарчимтай ажилладаг. JSF-ийн гол онцлогуудыг доорх гурван хэсэгт хуваан авч үзэж болно.

\subsubsection{1. Серверийн талын хөрвүүлэлт ба Facelets}
\hspace*{0.7cm}Хэрэглэгчээс хөтчөөр дамжуулан хүсэлт (HTTP Request) илгээх үед JSF нь тухайн хуудасны \texttt{Facelets} (\texttt{.xhtml}) загварын хуудсыг сервер дээр уншиж боловсруулдаг. Түүн доторх JSF-ийн тусгай тагуудыг (жишээлбэл \texttt{<h:inputText>}, \texttt{<h:dataTable>}) стандарт HTML, CSS болон шаардлагатай JavaScript код руу хөрвүүлэн угсардаг. Үүний дараа бүрэн боловсруулагдсан, эцсийн цэвэр HTML баримтыг хөтөч рүү буцаадаг (HTTP Response). Иймд хөтөч нь зөвхөн бэлэн болсон HTML-ийг зурж харуулах үүрэг хүлээх бөгөөд хуудасны үндсэн төлөв сервер дээр хадгалагддаг.

\subsubsection{2. Төлөвийн удирдлага ба Managed Beans}
\hspace*{0.7cm}JSF-ийн бас нэгэн гол онцлог нь UI элементүүд болон серверийн талын Java объектуудын (Managed Beans буюу Backing Beans) хооронд өгөгдлийн хоёр чиглэлт холбоос үүсгэдэг явдал юм. Хэрэглэгчийн оруулсан өгөгдөл эсвэл товч дарах үйлдэл нь шууд серверийн талд байгаа Java функцүүдтэй холбогдон ажжилладаг. Үүний тулд JSF нь тухайн хуудасны төлөвийг тусгай Component Tree бүтцээр санах ойд эсвэл хөтчийн нуугдмал талбарт хадгалж, дараагийн хүсэлт ирэх үед бүтцийг өмнөх байдалд нь сэргээж ажилладаг.

\subsubsection{3. Хүсэлт боловсруулах амьдралын мөчлөг}
\hspace*{0.7cm}Хэрэглэгчийн хийсэн үйлдэл бүрийг JSF нь маш нарийн тодорхойлогдсон 6 үе шаттай мөчлөгөөр боловсруулдаг. Үүнд:
\begin{itemize}
    \item Хүсэлт хүлээн авч, хуудасны Component бүтцийг сэргээх.
    \item Хэрэглэгчийн оруулсан утгуудыг хуудасны элементүүдэд олгох.
    \item Өгөгдлийн үнэн зөв байдал ба баталгаажуулалт хийх.
    \item Батлагдсан өгөгдлөөр серверийн объектын утгыг шинэчлэх.
    \item Хэрэглэгчийн үйлдлийг буюу үндсэн Java логикийг ажиллуулах.
    \item Эцсийн HTML баримтыг үүсгэж хөтөч рүү илгээх.
\end{itemize}

Дээр дурдсанчлан JSF нь аюулгүй байдал өндөртэй, сервер талд өгөгдлөө найдвартай удирддаг давуу талтай хэдий ч, UI хэсгийг динамикаар хурдан өөрчлөх эсвэл орчин үеийн баялаг интерактив үйлдэл нэмэх үед хуудсыг байнга дахин ачаалах шаардлага гарах зэрэг хязгаарлалттай тулгардаг. Тиймээс энэхүү төслийн архитектурт JSF-ийн бат бөх суурь болох өгөгдлийн хамгаалалт, чиглүүлэлтийг хэвээр үлдээж, хэрэглэгчийн өндөр интерактив үйлдэл шаардсан хэсгүүдэд микрофронтэндийг хөтөч дээр скрипт (\texttt{<script>}) байдлаар нэгтгэн суулгах архитектурын шийдлийг сонгосон болно.

\subsection{Webpack тохиргоо ба Интеграци}
Орчин үеийн JavaScript фрэймворкууд нь Webpack хэрэгсэл дээр суурилдаг бөгөөд төслийг ажиллуулах скриптүүдийг \texttt{package.json} дотор тохируулдаг. Зураг \ref{fig:package-update}-т үзүүлсний дагуу \texttt{"start"}, \texttt{"build"} скриптүүдийг ашиглан микрофронтэнд модулиудыг хөгжүүлэлтийн орчинд бие даан ажиллуулсан. Туршилтын загварт React модулиудыг JSF-тэй нэгтгэхдээ Webpack Module Federation-гийн оронд шууд багцлагдсан JavaScript файлын аргыг ашигласан. Энэ нь JSF хостын \texttt{.xhtml} хуудас нь хөтчид бүрэн ачаалагдсаны дараа \texttt{<script>} тагаар Webpack-аар үүсгэсэн \texttt{main.js} React модулиудын main.js файлыг дуудаж, \texttt{\#microfrontend-root} DOM элемент дотор React аппликэйшнийг зурах зарчмаар ажиллана. Энэ нь \ref{sec:integration-types}-д дурдсан интегра- цийн төрлүүдийн дагуу ажиллах үеийн интеграцийг ашигласан гэсэн үг юм. React модуль бүр бие даан угсрагддаг тул JSF хостын кодод нөлөөлөхгүйгээр тусдаа нэвтрүүлэх боломжтой бөгөөд энэ нь сул холбоост байдлыг хангана.

Микрофронтэнд хөгжүүлэлтэд түгээмэл ашиглагддаг Webpack Module Federation (WMF) аргачлалаас татгалзаж, Script Injection (буюу хөтчөөс \texttt{<script>} тагаар динамикаар дуудах) аргыг сонгосон нь техникийн болоод архитектурын тодорхой шалтгаантай. WMF нь үндсэндээ JavaScript дээр суурилсан хост болон JavaScript remote аппликэйшнүүдийн хооронд шууд холбогдож ажиллахад төгс зохицдог. Гэтэл энэхүү бүтээлийн үндсэн хост аппликэйшн нь серверийн талд HTML хуудсыг хөрвүүлэн гаргадаг Java буюу JSF технологи юм. Иймээс тэс өөр архитектуртай орчинд WMF-ийг хүчээр нэвтрүүлэх нь системийн тохиргоог хэт хүндрүүлж, нарийн төвөгтэй байдлыг үндэслэлгүйгээр ихэсгэх эрсдэлтэй. 

Харин үүний оронд Script Injection аргыг ашигласнаар JSF нь өөрийн \texttt{.xhtml} хуудсаа серверээс хэвийн рэндэрлэж гаргасны дараа, хөтөч нь React-ийн багцлагдсан \texttt{main.js} файлыг бие даан дуудаж ажиллуулж чадсан. Энэхүү арга нь хост болон микрофронтэнд хоорондын технологийн хамаарлыг хамгийн бага түвшинд барьж чадсан бөгөөд интеграцийн хувьд маш энгийн бөгөөд найдвартай архитектурын шийдэл болсон.

\subsection{CSS загварын давхцал}
Хэрэглэгч нэг микрофронтэндийн хуудаснаас нөгөө рүү шилжих үед хөтөч нь тухайн модулиудын CSS загваруудыг DOM-ийн \texttt{<head>} хэсэгт нэмж ачаалдаг бөгөөд ижил нэртэй CSS классууд нэгнийгээ дарж бичин, UI-ийн хэв гажилт үүсгэдэг байна. Энэ асуудлаас сэргийлэхийн тулд туршилтын загварт CSS Modules аргачлалыг ашигласан. Модуль бүрийн загварын файлыг \texttt{.module.css} өргөтгөлтэйгээр үүсгэж, \texttt{webpack.config.js} дотор \texttt{css-loader}-ын \texttt{modules} тохиргоог идэвхжүүлсэн. Webpack хөрвүүлэлтийн явцад CSS класс бүрийн нэрэнд \texttt{[name]\_[local]\_[hash:6]} загварын дагуу давтагдахгүй хэш утга залгадаг тул JSF үндсэн аппликэйшн болон бусад React модулиудын загвартай мөргөлдөхөөс бүрэн хамгаалагддаг.

Хэдийгээр CSS Modules нь React компонентүүдийн класс нэрийг дахин давтагдашгүй болгож, тэднийг бусад модуль болон JSF хэсэгтэй мөргөлдөхөөс бүрэн хамгаалдаг боловч нэгэн техникийн хязгаарлалт байдгийг дурдах нь зүйтэй. JSF хост аппликэйшний талд тодор- хойлогдсон глобал загварууд нь React компонентүүд рүү доошоо уламжлагдан орж ирж нөлөөлөх эрсдэл бий. Учир нь JSF болон React нь хөтөч дээр нэг л DOM модонд хамтдаа байрладаг. 

Энэ глобал нөлөөллөөс сэргийлж тусгаарлахын тулд Web Components стандартын нэг хэсэг болох \textit{Shadow DOM} ашиглах хувилбарыг онолын хувьд давхар судалсан. Гэвч Shadow DOM-ийг React-тэй хоршуулан ашиглах үед Webpack-ийн \texttt{style-loader}-ийн тохиргоог хэт хүндрүүлдэг төдийгүй, дэлгэцийн хүрээнээс гадуур дүрслэгддэг React Portals (жишээ нь Modal, Dialog, Tooltip) суурьтай 3-дагч UI компонентүүдийн загвар алдагдаж эвдрэх өндөр магадлалтай байдаг. Тиймээс хост болон микрофронтэндийн кодыг нэг дор удирдан хөгжүүлж буй энэхүү туршилтын нөхцөлд CSS Modules аргачлал нь архитектурын хувьд хавьгүй оновчтой бөгөөд хөнгөн шийдэл болно гэж дүгнэн сонгосон болно.

\subsection{Санал болгож буй шийдэл}
Уламжлалт системийг шинэчлэхэд үүсдэг дээрх хүндрэлийг шийдвэрлэхийн тулд Зураг \ref{fig:hybrid-mfe}-т үзүүлсэн холимог микрофронтэнд архитектурыг энэхүү судалгаагаар санал болгож байна. Шинээр хөгжүүлэх шаардлагатай модулиудыг React фрэймворкоор бие даан хөгжүүлж, JSF-ийн хуудсанд \texttt{<script>} тагаар динамикаар нэгтгэнэ. Ингэснээр хуучин системийн тогтвортой ажиллагааг хадгалахын зэрэгцээ, орчин үеийн технологийн давуу талыг ашиглан системийг үе шаттайгаар, эрсдэл багатайгаар шинэчлэх боломж бүрдэх юм.
