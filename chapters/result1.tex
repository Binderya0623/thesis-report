\section{Чанарын өгөгдөл}
Ялгаатай технологийн парадигмуудыг нэгтгэж нэгдсэн интерфэйс үүсгэх энэхүү туршилтын үед дараах үндсэн зарчмуудыг баримталж ажилласан. 

\begin{itemize}
    \item \textbf{Сул холбоост} - Туршилтын загварт үзүүлсэнчлэн, JSF болон React микрофронтэндүүд нь хоорондоо нарийн хамааралгүйгээр ажиллана. Микрофронтэндүүд нь ямар нэгэн төлөв эсвэл өгөгдлийг дундаа хуваалцдаггүй бөгөөд энэ нь тэдгээрийн хооронд нягт холбоо үүсэхээс бүрэн сэргийлдэг.
    \item \textbf{Нэг хариуцлагатай байх} - Туршилтын загвар нь хэд хэдэн микрофронтэндээс бүрдэх бөгөөд тэдгээр нь зөвхөн хариуцсан үүргийг гүйцэтгэнэ.
    \item \textbf{Бие даан нэвтрүүлэх боломжтой} - Серверийн талын JSF болон клиент талын React аппликэйшнүүдийг бие биеэсээ тусгаарлагдсан байдлаар хөгжүүлсэн. Микрофронтэнд аппликэйшн бүрд тусгайлсан домэйн болон репозитори ашиглан бие даан байршуулах боломжтой. 
    \item \textbf{Фрэймворкоос үл хамаарах} - Туршилтын загвар нь хоёр өөр архитектуртай JSF болон JavaScript фрэймворкуудыг үндсэн аппликэйшнд хамтад нь амжилттай ачаалж, уялдуу- лан ажиллуулж болохыг бодитоор харуулсан.
    \item \textbf{Нэмэлт хэрэгсэл ашиглах} - Зөвхөн React аппликэйшн нь микросервисийн зарчмыг бүрэн хангаж, JSF-тэй уялдуулахад хангалтгүй байсан тул Webpack хэрэгслийг оруулж ирсэн (Хавсралт А.1). Интеграцийг динамикаар хийхийн тулд Webpack-ийн нарийвчил- сан тохиргоо шаардагдсан.  React компиляци хийгдээгүй үедээ хөтөч дээр шууд ажиллах боломжгүй учраас Webpack ашиглаж микрофронтэнд бүрийг бие даасан \texttt{index.js}, \texttt{style.css} болгон багцалж, хуудсанд ачаалагдах үедээ \texttt{window.Module3.mount()} гэх мэтээр глобал хувьсагчаар дамжуулан ажиллахаар тохируулсан.
    \item \textbf{Асинхрон скрипт ачаалалт} - Хүлээн авагч аппликэйшнд JSF болон React скриптүүдийг нэгтгэн ачаалах үед үүссэн сорилтыг динамик дуудлага болон эхлүүлэх логик ашиглан амжилттай шийдвэрлэсэн.
    \item \textbf{CSS-тэй холбоотой асуудал} - Туршилтын үйл явцад тулгарсан гол сорилтуудын нэг нь өөр өөр микрофронтэндүүдийн загварууд нэг хуудсанд бие биеэ дарж бичигдэх эрсдэл байсан бөгөөд үүнийг CSS Modules ашиглан шийдвэрлэсэн.
\end{itemize}

Асуулгын төгсгөлийн хэсэгт багш нар болон хөгжүүлэгчдээс архитектурын харьцуулсан үнэлгээг тоон (1 - огт чухал биш, 2 - чухал байж болно, 3 - чухал, 4 - маш чухал) утгаар авч, ISO/IEC 25010 стандартын чанарын үзүүлэлтүүдийн харилцан нөлөөллийг тооцоолно.

\section{DEMATEL шинжилгээ}

Санал болгож буй JSF болон React (болон бусад JavaScript фреймворкууд) хосолсон микро- фронтэнд архитектурын хувьд программхангамжийн чанарын үзүүлэлтүүдийн хоорондын нарийн хамаарал, шалтгаан-үр дагаврын холбоог тогтоохын тулд 16 дахь асуулт буюу DEMATEL (Decision Making Trial and Evaluation Laboratory) аргачлалыг ашигласан. Судалгаанд оролцогчид сонгогдсон 7 чанарын үзүүлэлт нэг нэгэндээ хэрхэн нөлөөлж буйг 0-ээс 4 хүртэлх оноогоор үнэлсэн (0 - огт нөлөөгүй, 1 - бага зэрэг нөлөөлсөн, 2 - дундаж нөлөөтэй, 3 - их нөлөөтэй, 4 - маш өндөр нөлөөтэй). 

Энэ үнэлгээнд үндэслэн шууд нөлөөллийн матриц байгуулж, цаашлаад нийт харилцан хамаарлын матрицыг тооцоолон гаргана. Эндээс хоёр үндсэн утгыг гаргаж авна:
\begin{itemize}
    \item \textbf{Давамгайлал ($r+c$):} Энэ нь тухайн чанарын үзүүлэлтийн систем дэх нийт ач холбогдлыг илэрхийлнэ. Утга өндөр байх тусам уг архитектурт хамгийн чухал үүрэгтэй үзүүлэлт гэж үзнэ.
    \item \textbf{Хамаарал ($r-c$):} Энэ утга нь тухайн үзүүлэлтийг шалтгаан эсвэл үр дагавар болохыг тодорхойлно. Эерэг утгатай ($r-c > 0$) бол бусад үзүүлэлтүүдээ чиглүүлдэг \textit{шалтгааны бүлэгт}, сөрөг утгатай ($r-c < 0$) бол бусдаасаа хамаардаг \textit{үр дагаврын бүлэгт} хамаарна.
\end{itemize}

\textit{[Дүгнэлтийн загвар: Судалгааны үр дүн гарсны дараа доорх хэсгийг нөхөж бичнэ]}

Хүснэгтийн үр дүнгээс харахад $r+c$ утгаар [Хамгийн өндөр оноотой QAs] нь холимог микрофронтэнд архитектурын хамгийн чухал үзүүлэлтүүд болох нь тогтоогдсон. Мөн $r-c$ утгын хувьд [Шалтгаан болсон QAs] зэрэг нь эерэг утгатай гарсан тул шалтгааны бүлэгт багтаж байна. Өөрөөр хэлбэл, JSF болон React-ийг уялдуулж хөгжүүлэхэд эдгээр чанарууд нь анхдагч болж бусад үзүүлэлтүүдээ сайжруулж байна. Харин [Үр дагавар болсон QAs] нь сөрөг утгатай гарсан бөгөөд эдгээр нь дээрх шалтгааны үр дүнд сайжирч буй үр дагаврын бүлэгт хамаарч байна.

\section{Судалгааны хязгаарлалт}

Энэхүү судалгааны ажил нь цоо шинээр хөгжүүлэгдэж буй холимог архитектурын шийд- лийг бодитоор хэрэгжүүлж, үнэлсэн ач холбогдолтой хэдий ч дараах хэд хэдэн хязгаарлалттай бөгөөд цаашдын судалгаанд анхаарах шаардлагатайг дурдах нь зүйтэй.

\begin{enumerate}
    \item \textbf{Оролцогчдын тоо болон субъектив үнэлгээ:} Кэйс судалгааны өгөгдлийг хөгжүүлэл- тийн багийн болон холбогдох инженерүүдээс бүрдсэн харьцангуй цөөн тооны оролцогч- доос цуглуулахаар төлөвлөж байна. Мөн DEMATEL аргачлалын оролтын өгөгдөл нь оролцогчдын мэдлэг, туршлагад үндэслэсэн субъектив үнэлгээ тул үр дүнд тодорхой хэмжээний хэлбэлзэл байх магадлалтай. Монгол улсад микрофронтэнд архитектурын туршлагатай хөгжүүлэгчдийн тоо хязгаарлагдмал байгаа нь үүнд нөлөөлж болзошгүй.
    
    \item \textbf{Турших загвар:} Энэ судалгаагаар МУИС-ийн Дипломын ажлын удирдлагын системд зориулан хөгжүүлсэн холимог микрофронтэнд архитектур нь одоогоор эцсийн хэрэглэгч- дэд бүрэн нэвтрээгүй, турших загвар түвшинд хэрэгжсэн болно. Иймд бодит үйлдвэрл- элийн орчинд их хэмжээний хандалт авах үеийн гүйцэтгэлийн нарийвчилсан өгөгдлийг цуглуулах боломжгүй байв. 
    
    \item \textbf{Хөгжүүлэгчийн өнцөг:} Системийн архитектур болон чанарын үнэлгээг зөвхөн хөгжүү- лэлтийн үйл явц, инженерчлэлийн өнцгөөс (засварлах боломжтой байдал, модульлаг байдал г.м) үнэлсэн. Эцсийн хэрэглэгчдийн (оюутан, багш нар) хэрэглэхэд хялбар байдал, сэтгэл ханамж, UI/UX-ийн хэрэглэгчийн туршлагын судалгааг энэхүү ажлын хүрээнд гүйцэтгээгүй бөгөөд энэ нь цаашдын судалгааны сэдэв болох бүрэн боломжтой.
\end{enumerate}
